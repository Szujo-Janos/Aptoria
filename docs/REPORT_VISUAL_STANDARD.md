# Aptoria Report Visual Standard

## Purpose

Every Aptoria report export must look like a professional, printable QA evidence document, not like a raw dashboard fragment.

This rule is mandatory for every current and future report type:

- Full project QA report
- Release readiness report
- Release decision report
- Evidence package summary
- Technical QA summary
- Any future report/export that represents fixed QA evidence

## Source of truth

The shared implementation is:

```text
app/Services/ReportVisualStandardService.php
```

Do not hand-code a separate HTML wrapper inside controllers, report builders or portal download actions.

Every HTML report download must pass through:

```php
ReportVisualStandardService::exportHtml(ReportVersion $report)
```

Markdown fragments may still be generated by domain-specific builders, but their HTML representation must be wrapped by the shared standard.

## Required visual structure

Every HTML report must use this top-level structure:

```html
<main class="report" data-aptoria-report-standard="report-visual-standard-v1.1">
  <header class="report-header">...</header>
  <div class="summary-strip">...</div>
  <div class="report-body">...</div>
  <footer class="report-footer">...</footer>
</main>
```

Required sections:

1. Report header
2. Metadata table
3. Summary strip
4. Evidence notice
5. Numbered report sections
6. Footer with report standard version and checksum reference

## Header rules

The report header must contain:

- Aptoria brand area
- confidentiality / evidence label
- report title
- short report subtitle
- prepared-by identity
- role/title
- organization
- generation timestamp

The layout must remain usable in print and small screens.

## Metadata table rules

Every report must include a `meta-table` with at least:

- Project
- Report type
- Base URL or target context
- Report status
- Generated at
- Checksum

Checksums must be shown in a monospace/code style and must wrap safely.

## Summary strip rules

Every report must include a five-cell `summary-strip`.

Preferred order:

1. Report status
2. Release/readiness score
3. Blockers
4. Warnings
5. Evidence count or open finding count fallback

Tone classes:

- `ok` for acceptable / approved / zero-blocker values
- `warn` for review-needed values
- `danger` for blocker / failed / critical values

The summary strip must never become a full-width badge bar. It is a compact KPI strip.

## Body rules

Report body must use:

- `.notice` for report/evidence caveat
- `.report-section` for every major block
- numbered section markers inside section headings
- simple tables with full width
- readable, printable text density
- no dashboard cards, no DataTables, no action buttons

Report content is evidence, not an interactive admin UI.

## Table rules inside reports

Report tables must:

- use full width
- collapse borders
- use light grey table headers
- use compact padding
- avoid fixed pixel hacks
- allow text wrapping
- remain printable

Long values such as endpoint paths, checksums and errors must wrap instead of overflowing.

## Typography rules

HTML reports use:

```css
font-family: Arial, Helvetica, sans-serif;
font-size: 13px;
line-height: 1.48;
```

The report design intentionally differs from the admin dashboard because it is an exported document.

## Print rules

Every report must include print CSS:

- white page background
- no outer browser-style margin frame
- report sections should avoid breaking inside when possible
- links inherit text color when printed

## Responsive rules

On narrow screens:

- header columns stack
- prepared-by block aligns left
- summary strip becomes two columns
- footer becomes one column

## Forbidden patterns

Do not use these in exported HTML reports:

- Dashboard cards as report structure
- Admin panel headers
- action buttons
- DataTables
- dropdowns
- SweetAlert hooks
- Bootstrap toast hooks
- unrelated sidebar/topbar markup
- raw unwrapped `content_html` as downloadable HTML
- per-report custom CSS wrappers

## Developer rule for new reports

When adding a new report type:

1. Create the domain-specific report data.
2. Generate Markdown or a safe HTML fragment.
3. Store the content in `report_versions`.
4. Export HTML only through `ReportVisualStandardService`.
5. Add or update feature coverage that checks for `data-aptoria-report-standard="report-visual-standard-v1.1"`.

## QA checklist

A report/export change is not complete until this is verified:

- HTML download opens as a standalone document.
- Header, metadata table, summary strip and footer are present.
- Report uses `data-aptoria-report-standard="report-visual-standard-v1.1"`.
- Checksum is visible.
- Approval/sign-off metadata appears when available.
- Tables do not overflow the page.
- Print preview is readable.
- Mobile width stacks correctly.
- Public client portal report downloads use the same standard.


## Hotfix addendum — v1.1

The shared report shell must embed the real Aptoria SVG logo as a standalone `data:image/svg+xml;base64,...` image. A one-letter placeholder mark is not acceptable for generated reports.

The generated HTML must not expose unresolved translation keys such as `messages.projects.project`. Missing translations must fall back to clean human-readable labels.

The report content body must not introduce a second oversized document title. Any stored inner `h1`/`h2` content must be demoted inside the Technical Appendix so the report header remains the only primary title.


## Official logo asset

All Aptoria surfaces and generated reports must use the single official SVG logo asset:

```text
public/assets/aptoria-ui/assets/images/logo-color.svg
```

Deprecated PNG logo files are not allowed in the application shell, setup loader, auth screens, client portal or generated report exports. Report HTML exports must embed this SVG as `data:image/svg+xml;base64,...` so downloaded reports remain standalone and branded. One-letter placeholder marks are not acceptable.

## Evidence Pack export addendum — v0.0.45

Evidence Pack HTML downloads are report exports and must not use a separate raw wrapper.

Mandatory implementation:

```text
App\Services\ReportVisualStandardService::exportEvidencePackHtml(EvidencePack $pack)
```

Evidence Pack PDF downloads must be generated from the same fixed EvidencePack record and must include, at minimum:

- report standard version
- project name and base URL
- pack type and status
- generated timestamp
- checksum
- selected sections
- evidence/finding/import/contract/risk counts
- linked readiness/report context when available

ZIP packages must include the standardized `report.html` and `report.pdf` files in addition to README/Markdown, manifest JSON and checksum files.

The Evidence Pack detail screen should offer HTML, PDF and ZIP as primary export actions, while keeping Markdown and JSON available from list actions.

## Evidence PDF logo addendum — v0.0.45 hotfix

The dependency-free Evidence Pack PDF renderer must not draw a temporary `APT` placeholder mark. It must render the real Aptoria brand in the PDF header.

The PDF renderer may use the PDF-safe raster derivative below because the custom PDF writer does not natively embed SVG images:

```text
public/assets/aptoria-ui/assets/images/logo-color-pdf.jpg
```

This file is not an independent logo source. It is a generated PDF embedding derivative of the official SVG asset:

```text
public/assets/aptoria-ui/assets/images/logo-color.svg
```

If the official SVG changes, the PDF derivative must be regenerated from that SVG. The old deprecated PNG logo files remain forbidden.
